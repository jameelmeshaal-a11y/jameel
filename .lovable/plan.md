

# تحليل وإصلاح البنود غير المسعّرة + تحسين واجهة مكتبة الأسعار

## السبب الجذري: خلل في تطبيع الوحدات (Unit Normalization)

### المشكلة الرئيسية
دالة `normalizeUnit()` في `similarItemMatcher.ts` لا تتعامل مع عدة صيغ وحدات موجودة فعلياً في البيانات:

```text
المكتبة فيها:  م²  (70 بند) + م2 (20 بند) + م³ (14 بند) + م3 (10 بند)
                م. .ط (9 بنود) + م ط (2 بند) + مجموعة (0 بند!)

الـ regex الحالي:  /م2|m2/g → "m2"   ← يمسك "م2" لكن لا يمسك "م²" (Unicode ²)
                   /م3|m3/g → "m3"   ← يمسك "م3" لكن لا يمسك "م³" (Unicode ³)
                   /م\.ط/g  → "ml"   ← لا يمسك "م. .ط" أو "م ط"
```

**النتيجة**: بند في الـ BoQ وحدته `م²` وبند في المكتبة وحدته `م2` — لا يتطابقان أبداً لأن `normalizeUnit` ينتج قيمتين مختلفتين. والمحرك يعمل `continue` (تخطي) عند عدم تطابق الوحدة.

### مشكلة ثانوية: الأوصاف الطويلة جداً
البنود المدمجة (merged descriptions) تصل لمئات الكلمات — وهذا يُخفّض نسبة Jaccard لأن `intersection/union` تصغر كلما زاد عدد الكلمات. الحل: استخدام الجزء الأخير بعد `—` كوصف أساسي للمطابقة أيضاً.

### مشكلة ثالثة: وحدات غير موجودة في المكتبة
بعض البنود تستخدم `مجموعة` و`بند` — المكتبة فيها `بند` (22 بند) لكن ليس فيها `مجموعة`. هذه تحتاج إما إضافة بنود بالوحدة المناسبة أو تطبيع `مجموعة` → `set/بند`.

## الحل

### 1. إصلاح `normalizeUnit()` — `src/lib/pricing/similarItemMatcher.ts`

توسيع الدالة لتغطي كل الصيغ الموجودة فعلياً في البيانات:

```text
م² / م٢ → m2
م³ / م٣ → m3  
م. .ط / م .ط / م ط → ml
مجموعة / set / مجم → set
بند / lump / ls → lump
كجم / kg → kg
طن / ton → ton
```

### 2. تحسين المطابقة للأوصاف الطويلة — `src/lib/pricing/matchingV3.ts`

إضافة منطق يستخرج الجزء الأخير بعد `—` من الوصف المدمج ويستخدمه كوصف مختصر إضافي للمقارنة. هذا يرفع الدقة للبنود التي وصفها الأصلي قصير لكن السياق المدمج طويل جداً.

### 3. تحسين واجهة مكتبة الأسعار — `src/pages/RateLibraryPage.tsx`

- نقل مربع البحث إلى الأعلى بجانب العنوان بمساحة أكبر
- وضع الفئات في شريط قابل للتمرير (scrollable) بدلاً من التفاف على عدة أسطر
- إضافة عدّاد نتائج البحث
