

# مشكلة: جميع بنود البلوك بنفس السعر (201.25) رغم اختلاف السُمك

## التشخيص

### ما حدث بالضبط
المكتبة تحتوي على **سجل بلوك واحد فقط** — "جدار بسمك 100 مم" بسعر 201.25 ريال/م². جميع بنود البلوك (100مم، 150مم، 200مم، مقاوم للحريق) تم ربطها بهذا السجل الوحيد بثقة 95-99%.

### السبب الجذري — خلل مزدوج

**1. خلل في محلل الأبعاد (Parser Bug):**
الـ Regex الحالي للسُمك هو:
```
/(?:سمك|thickness|thk)\s*(\d+)/
```
لكن النص في جدول الكميات يقول **"بسمك 150 مم"** (بحرف الباء). المحلل لا يتعرف على "بسمك" لأنه يبحث عن "سمك" فقط. النتيجة: **لا يكتشف أي سُمك** → لا مقارنة → لا تعارض → يقبل المطابقة.

**2. نقص في بيانات المكتبة:**
لا يوجد سجل مستقل لجدار 150مم أو 200مم أو مقاوم للحريق. حتى لو أصلحنا المحلل، المحرك سيرفض المطابقة ولن يجد بديلاً — وهو السلوك الصحيح (أفضل من سعر خاطئ).

## الحل — 3 إجراءات

### 1. إصلاح محلل السُمك — `synonyms.ts`
توسيع regex ليشمل "بسمك" و"سماكة" و"سماكه":
```typescript
// قبل: /(?:سمك|thickness|thk)\s*(\d+)/
// بعد: /(?:بسمك|سمك|سماكة|سماكه|thickness|thk)\s*(\d+)/
```

### 2. تفعيل Hard Gate للسُمك — `matchingV3.ts`
حالياً الـ Hard Gate يعمل فقط على أبعاد WxH. يجب توسيعه ليشمل نوع `thickness` أيضاً:
```text
إذا كان البند والمطابقة كلاهما يحتوي سُمك مختلف → تخطي كامل (score = 0)
مثال: بند "بسمك 150 مم" vs مكتبة "بسمك 100 مم" → ⛔ تخطي
```

### 3. إعادة تعيين البنود المتضررة — Database
إزالة الربط الخاطئ من بنود البلوك (150مم، 200مم، مقاوم حريق) وإعادتها لـ `pending` ليُعاد تقييمها. بند 100مم صحيح ويبقى كما هو.

## الملفات المتأثرة

| الملف | التغيير |
|---|---|
| `src/lib/pricing/synonyms.ts` | إصلاح regex السُمك: إضافة "بسمك"، "سماكة" |
| `src/lib/pricing/matchingV3.ts` | توسيع Hard Gate ليشمل `thickness` بجانب `dimensions` |
| Database | إعادة تعيين بنود البلوك ذات السُمك المختلف عن 100مم |

## ما لا يتأثر
- بند بلوك 100مم — مطابقته صحيحة ولن تتغير
- أي بنود مسعّرة سابقاً بشكل صحيح
- باقي منطق المطابقة والاعتماد

## ملاحظة مهمة
بعد التنفيذ، بنود 150مم و200مم والمقاوم للحريق ستظهر بدون سعر (`pending`) — تحتاج إضافة سجلات جديدة في المكتبة بالأسعار الصحيحة لكل سُمك.
