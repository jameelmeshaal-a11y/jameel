

# بناء نظام مطابقة "يفهم" بدلاً من "يحفظ" — تحسين شامل

## المشكلة الجوهرية

النظام الحالي يعتمد على regex ثابت للتعرف على المواصفات الفنية (سُمك، قطر، أبعاد). أي تنويع لغوي بسيط ("بسمك" vs "سمك" vs "بالسمك") يكسر التعرف. المطلوب: نظام يحلل السياق الكامل ويفهم المعنى.

## الحل — 4 طبقات

### 1. محلل أبعاد ذكي (Smart Dimension Parser) — `synonyms.ts`

بدلاً من regex واحد لكل مصطلح، ننقل إلى نظام **تطبيع قبل المطابقة**: إزالة كل البادئات العربية (ب، ال، بال، وال) من النص قبل تشغيل regex الأبعاد. هذا يعني أن "بسمك"، "والسمك"، "بالسمك"، "للسمك" كلها تتحول تلقائياً إلى "سمك" قبل الفحص — بدون حفظ كل تنويع يدوياً.

```text
قبل:  /(?:بسمك|سمك|سماكة|سماكه|thickness|thk)\s*(\d+)/
بعد:  1) تطبيع النص: إزالة بادئات (ب، ال، و، بال، وال، لل)
      2) regex بسيط: /(?:سمك|سماك[ةه]|thickness|thk)\s*(\d+)/
      → يلتقط تلقائياً: بسمك، والسمك، بالسمك، للسمك، وسمك...
```

نفس المنهج لـ "قطر" (بقطر، والقطر)، "مقاس" (بمقاس، والمقاس)، إلخ.

### 2. تطبيع ذكي للنص (Deep Normalization) — `synonyms.ts`

إضافة دالة `deepNormalize(text)` تُستخدم في `detectConcepts` وفي `parseDimensions`:

```text
- إزالة البادئات العربية: ب، ال، و، بال، وال، لل، ف، ك
- تطبيع الهمزات والتاء المربوطة (موجود)
- إزالة التشكيل (موجود)
- إزالة علامات الترقيم والأقواس
- تطبيع المسافات المتعددة
```

هذا يجعل النظام يتعرف على أي بادئة عربية تلقائياً بدون إضافة كل حالة يدوياً.

### 3. مراجعة وتوسيع مكتبة المرادفات — `synonyms.ts`

بناءً على فحص مكتبة الأسعار (893 بند، 46 فئة)، إضافة مفاهيم جديدة ناقصة:

```text
إضافات مطلوبة:
- "بلوك_100مم" / "بلوك_150مم" / "بلوك_200مم" — تفريق بالسُمك
- "بلوك_مقاوم_حريق" — نوع خاص
- "عزل_مائي" / "عزل_حراري" — لمنع الخلط
- "لياسة" / "محارة" — مرادفات
- "رخام" / "جرانيت" / "بورسلان" — أنواع تشطيب
- "مضخة" / "خزان" — معدات ميكانيكية

+ أزواج Anti-Confusion لكل مجموعة جديدة
```

### 4. فحص شامل للبنود المسعّرة حالياً — Database Audit

تنفيذ استعلام يكشف **جميع** البنود المشبوهة في كل المشاريع:

```sql
-- بنود نفس الفئة + نفس الوحدة + أوصاف مشابهة لكن أسعار مختلفة جذرياً
SELECT bi.description, bi.unit_rate, rl.standard_name_ar, rl.target_rate,
       bi.unit_rate / NULLIF(rl.target_rate, 0) as ratio
FROM boq_items bi
JOIN rate_library rl ON bi.linked_rate_id = rl.id
WHERE bi.unit_rate > 0
  AND ABS(bi.unit_rate - rl.target_rate) / NULLIF(rl.target_rate, 1) > 0.5
ORDER BY ratio DESC;
```

إعادة تعيين أي بنود مشبوهة لـ `pending`.

## الملفات المتأثرة

| الملف | التغيير |
|---|---|
| `src/lib/pricing/synonyms.ts` | إضافة `deepNormalize()` + إعادة بناء `parseDimensions` بتطبيع ذكي + مرادفات جديدة |
| `src/lib/pricing/matchingV3.ts` | لا تغيير (يستفيد تلقائياً من تحسينات synonyms) |
| Database | فحص شامل + إعادة تعيين البنود المشبوهة |

## ما لا يتأثر
- خوارزمية التقييم الأساسية (scoreCandidate) — بدون تغيير
- حواجز الحماية الحالية (Magnitude Guard, Sub-item Guard) — تبقى كما هي
- البنود المعتمدة يدوياً (manual_override) — محمية ولن تتأثر

## النتيجة المتوقعة
- "بسمك 150مم" و"سمك 150 مم" و"بالسمك 150" كلها تتعرف كـ thickness=150 تلقائياً
- لا حاجة لإضافة كل تنويع لغوي يدوياً — النظام يطبّع أولاً ثم يحلل
- Anti-Confusion يمنع خلط بلوك 100مم مع 150مم أو 200مم
